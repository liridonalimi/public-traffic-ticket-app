# Technical: Bluetooth Ticket Printing

## Scope

This module connects a stored ticket sale to a physical Bluetooth Classic label or receipt printer that supports ESC/POS over the serial-port profile (SPP). It also includes a PDF test printer for validating ticket content without hardware. Both outputs print immediately after a sale and use the same durable status and retry workflow.

## Sale and print sequence

1. The driver selects a paired printer.
2. The app creates and persists the ticket with `PENDING` print status.
3. The app builds the printable ticket from the sale, bus, route, fare, price, and timestamp.
4. `BluetoothEscPosTicketPrinter` opens an RFCOMM connection using the standard SPP UUID.
5. The adapter sends an ESC/POS initialization command, ticket text, feed, and cut command.
6. The stored ticket becomes `PRINTED` on success or `FAILED` with the device error on failure.

The shift cannot close while a ticket is pending or failed. Retrying reuses the existing ticket ID and increments `printAttempts`; it never records another sale.

## PDF test output

`PdfTicketPrinter` is always available, even when Bluetooth permission is denied. It generates a narrow, 58 mm-style PDF in the app-specific documents directory and returns its path with the successful print result. The UI shares the most recent file with an installed PDF viewer through Android `FileProvider`.

The preview follows the visual hierarchy of a Kosovo receipt: operator heading, Albanian ticket title, fare/quantity/price row, total, cash payment, sale metadata, and a QR code. During development it is deliberately marked `KUPON TESTUES - JO FISKAL`. The QR contains only local test-ticket data and is not an ATK verification code. Fiscal identifiers, VAT declarations, the fiscal logo, and ATK digital signatures must not be added until an authorized fiscalization integration supplies them.

PDF output is a functional validation substitute, not a hardware simulation. It verifies the sale payload, ticket layout, persisted print success, fare totals, and shift rules. It does not verify Bluetooth transport, ESC/POS interpretation, cutter or feed behavior, paper width, speed, or printer reliability.

## Persistence

Each ticket stores:

- `printStatus`: `PENDING`, `PRINTED`, or `FAILED`
- `printAttempts`
- `lastPrintError`

Older local tickets default to `PENDING`. The selected printer name and Bluetooth MAC address are also retained in `SharedPreferences`.

## Android permissions

Android 12 and newer require the runtime `BLUETOOTH_CONNECT` permission to inspect paired devices and connect. Earlier versions use the legacy `BLUETOOTH` and `BLUETOOTH_ADMIN` manifest permissions.

## Compatibility boundary

The included byte encoder is ESC/POS. It does not claim compatibility with printers requiring ZPL, TSPL, CPCL, BLE GATT, USB, or a proprietary vendor SDK. Add another `TicketPrinter` implementation for those transports and command languages.

## Verification

`EscPosTicketTest` verifies that the generated byte stream includes initialization/cut commands, the non-fiscal test marker, ticket code, bus, route, operator, fare, formatted total, and sale time. `PdfTicketContentTest` verifies that every required ticket field, formatted price, and explicitly non-fiscal QR payload are passed to PDF rendering. The Android debug unit-test build verifies both printer integrations compile with the app.

Physical acceptance testing still requires the target printer model because paper width, character tables, cutter support, and command-language compatibility are hardware-specific.
